# 要求定義

## 1. 機能要求

### 1.1 メンバー管理

- メンバーの氏名を登録できる
- メンバーに部署と管理者を紐付けられる
- 部長・マネージャーは直属メンバーおよび自部署メンバーを閲覧・編集できる
- メンバーは論理削除できる
- 削除済みメンバーはシステム運用者が復元できる

### 1.2 案件経験管理

- 案件経験はメンバー単位で個別登録する
- 同一案件に複数メンバーが参加していても共通案件マスタは利用しない
- 案件名、期間、概要、顧客名、業界、状態を保持できる
- 案件ごとに複数の報告を時系列で登録できる
- 報告種別は定期、終了、任意を持つ
- 案件終了時と定期更新の両方に対応する

### 1.3 Markdown取り込み

- 入力ファイルはMarkdown形式のみとする
- 1ファイルは1メンバー・1案件・1報告とする
- アップロード時に部長が対象メンバーと対象案件を画面で選択する
- Markdown内のメンバー名や案件名は確認用とする
- 解析結果は原則として自動保存する
- 解析失敗または項目判定が曖昧な場合に警告する
- 正常に解析できた項目は警告があっても保存する
- 元Markdownの保存有無はシステム設定で切り替えられる

### 1.4 人物評価

- 人物評価は時系列で蓄積する
- 定期評価、案件終了評価、任意評価を登録できる
- 案件終了評価は対象案件へ任意で紐付けられる
- 評価項目は固定項目と自由記述を併用する
- 数値評価やランキングを必須としない

### 1.5 スキル管理

- スキルは上司入力とAI抽出の両方に対応する
- AI抽出スキルは候補として保存する
- 上司が候補を修正、確認、却下できる
- スキルには案件報告、評価、上司コメントなどの根拠を紐付ける
- スキルレベルのスコアリングは行わない

### 1.6 AI案件分析

- 案件登録・更新時に自動実行しない
- 上司が任意のタイミングで実行する
- 分析結果は先に保存し、上司が後から修正できる
- 実行日時、モデル、プロンプトバージョン、参照データを記録する
- 分析結果に根拠を持たせる

### 1.7 推薦プロジェクト

- 推薦対象メンバーを選択できる
- 推薦目的、推薦先、要件、強調点、文体、出力形式を入力できる
- 推薦文のAIドラフトを生成できる
- 推薦文生成時は毎回、最新データを分析する
- AI生成文と根拠を段落単位で紐付ける
- 根拠不足は警告する
- 警告があっても生成・編集・確定を妨げない
- 上司はAIドラフトを編集できる
- 推薦文をバージョン管理できる
- 最終版を確定できる

### 1.8 監査・履歴

- 業務データの作成、更新、論理削除、復元を監査ログに記録する
- AI分析結果の修正も記録する
- AI設定変更も記録する
- Markdown取り込みによる自動反映も記録する
- 変更前後のデータをJSONで保持する

### 1.9 データ保持

- 業務データは原則として論理削除する
- 会社または環境ごとに保持期間を設定できる
- 保持期限到達後も自動で即時物理削除しない
- 物理削除はシステム運用者のみ実行できる
- 物理削除前に対象と関連データを確認できる
- 物理削除操作の記録を残す

### 1.10 AIモデル管理

- AI Gatewayを利用する
- OpenAI互換APIを利用する
- Custom ProviderとしてAI Gatewayへ接続する
- AIモデル、接続先、認証情報、生成パラメータをシステム運用者が設定できる
- 一般利用者はAIモデルを変更できない

## 2. プロトタイプ環境要件

- 閉じた社内ネットワークで動作する
- インターネットへ公開しない
- DB接続情報、AI Gateway URL、APIキー、モデル名は `.env` で管理する
- `.env` はGit管理対象外とする
- `.env.example` にはキー名のみを記載し、実値を含めない
- 初期版では企業IdP、OIDC、SAML、Vault、Secret Managerを必須としない
- 認証が必要な場合は簡易ログインまたは社内リバースプロキシ配下での利用を想定する
- 外部公開へ移行する際はセキュリティ要件を再定義する

## 3. AI利用原則

- AIは事実を創作しない
- 根拠のない推論を出力しない
- 推論を行う場合は根拠を表示する
- AIは推薦可否を決定しない
- AIは人材を評価・順位付けしない
- AIは不足情報を警告する
- 警告は業務判断を阻害しない
- 最終責任は上司にある

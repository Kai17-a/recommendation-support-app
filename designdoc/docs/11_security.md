# セキュリティ設計

## 前提

本設計は、**閉じた社内ネットワークで利用するプロトタイプ**を対象とする。

- インターネットへ公開しない
- 不特定多数の利用者を想定しない
- 業務適合性とAI機能の検証を優先する
- セキュリティは必要最低限とする

本番展開や外部公開を行う場合は、別途セキュリティ設計を実施する。

## 接続情報

以下は `.env` で管理する。

- DB接続情報
- AI Gateway Base URL
- AI Gateway APIキー
- 利用モデル名
- タイムアウト、リトライ設定
- アプリケーション用秘密値

### 最低限のルール

- `.env` をGitへコミットしない
- `.gitignore` に `.env` を登録する
- `.env.example` を用意する
- `.env.example` に実際のAPIキーやパスワードを記載しない
- APIキーやDB接続文字列をログへ出力しない

## 認証

プロトタイプでは次のいずれかを採用する。

- 簡易なアプリケーションログイン
- 社内リバースプロキシによるアクセス制限
- 閉域内利用者を信頼し、暫定的に認証を省略

認証を省略する場合でも、監査ログ上の操作者を識別できるようにする。

企業IdP、OIDC、SAML、多要素認証は初期版の必須要件としない。

## 認可

- 部長・マネージャー
- システム運用者

部長・マネージャーは直属メンバーおよび自部署メンバーを対象とする。システム運用者は設定、監査、復元、物理削除を担当する。

## Markdown入力

- `.md` のみ許可
- ファイルサイズ上限を設定
- テキストファイルであることを確認
- 外部URLや画像を自動取得しない
- Markdown本文をAIへの命令ではなくデータとして扱う
- AI出力をJSON Schemaで検証する

## 通信

閉域ネットワーク内でも容易ならHTTPSを利用する。ローカル開発や単一端末での検証ではHTTPを許容する。

## 本番化時に再検討する項目

- OIDCまたはSAML
- 多要素認証
- Secret ManagerまたはVault
- TLS証明書管理
- 保存データ暗号化
- 脆弱性診断
- WAFとレート制限
- 個人情報のマスキング

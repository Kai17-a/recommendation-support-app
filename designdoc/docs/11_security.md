# セキュリティ設計

## 1. 認証

- 企業のIdP連携を推奨する
- OIDC DiscoveryとJWKSによって署名、発行者、Audience、有効期限を検証する
- APIでは短命Bearerアクセストークンを利用する
- OIDCの`sub`を利用者に紐付ける。初回のみ、IdPが検証済みとしたメールアドレスで既存利用者を紐付ける
- IdPクライアントシークレットはAPIで保持せず、APIはアクセストークンの検証のみを行う
- システム運用者には多要素認証を推奨する

## 2. 認可

### 部長・マネージャー

閲覧・編集可能範囲:

- 自分の直属メンバー
- 自部署のメンバー

他部署のメンバーは原則閲覧不可。

### システム運用者

- システム設定
- AI設定
- 全体監査
- 復元
- 物理削除

業務上の人物評価本文を常に閲覧できるかは、企業ポリシーに応じて分離可能とする。

## 3. AI Gateway送信制御

- 必要な情報のみ送信する
- システム内部IDを外部向け表示名へ置換可能にする
- 不要な個人情報をプロンプトへ含めない
- 認証情報をプロンプトへ含めない
- 送信内容を監査可能にする
- AI Gateway側のログ保持方針を確認する

## 4. 秘密情報管理

以下をDBへ平文保存しない。

- AI Gateway APIキー
- IdPクライアントシークレット
- 暗号鍵

## 5. Markdown安全対策

- `.md` 以外を拒否する
- ファイルサイズ上限を設ける
- バイナリ判定を行う
- HTMLを許容する場合はサニタイズする
- 外部画像や外部リンクの自動取得を行わない
- Markdown内の命令文をシステムプロンプトより優先しない

## 6. プロンプトインジェクション対策

Markdown本文は信頼できない入力として扱う。

- 「以前の指示を無視せよ」等を命令として実行しない
- データと命令を明確に区切る
- 構造化出力を必須にする
- AI出力をスキーマ検証する
- ツール実行権限をLLMへ直接与えない

## 7. 監査

- 閲覧ログの要否は企業ポリシーで決定する
- 更新、削除、復元、物理削除、AI設定変更は必須記録
- 監査ログ自体の更新・削除権限を制限する

## 8. 暗号化

- 通信はTLS
- DB、バックアップ、保存ファイルは保存時暗号化
- 秘密情報は専用Secret管理
